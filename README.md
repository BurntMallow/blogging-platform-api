# Blogging Platform API

![Coverage](.github/badges/jacoco.svg)

Containerized REST API for a Blogging Platform built with Spring Boot 4 and PostgreSQL

---

## Quick Start (Run with Docker)

### Prerequisites
* [Docker Desktop](https://www.docker.com/products/docker-desktop/) (includes Docker Compose)
* Postman, a browser, or `curl` (for testing endpoints)

### Spin Up the Application
The project utilizes a multi-stage Docker build. This means Java compilation, dependency downloading, and environment isolation happen entirely inside Docker. You do **not** need Java or Maven installed on your host machine to run this project.

1. Clone the repository:
```bash
git clone https://github.com/burntmallow/blogging-platform-api.git
cd blogging-platform-api
```

2. Build and start the API and PostgreSQL database simultaneously:
```bash
docker compose up --build
```

The API will automatically compile and spin up at http://localhost:8080.
* **Health Check Probe:** http://localhost:8080/actuator/health

> **Tip:** This command streams the application logs directly to your terminal window. To shut down the application and database containers safely at any time, simply press `Ctrl + C` inside this terminal. If you want to wipe the database and reset your practice data, run `docker compose down -v` afterward.

## Tech Stack & Architecture

* Backend Framework: Spring Boot 4.x (Java 21)
* Data Access: Spring Data JPA (Hibernate)
* Database: PostgreSQL 16
* Containerization: Docker & Docker Compose

### Architecture Notes
The project follows a standard Layered Architecture:
Controller (Web) -> Service (Business Logic) -> Repository (Data Access) -> PostgreSQL

---

## Database Lifecycle & Initialization

This project utilizes **Spring Data JPA (Hibernate)** to manage the PostgreSQL schema. 

* **Development Strategy:** The application profile uses `spring.jpa.ddl-auto=update`. When Docker Compose spins up the containers, Hibernate automatically inspects the Java `@Entity` classes and generates/updates the PostgreSQL database tables dynamically.
* **Production Note:** While `update` is used here for rapid prototyping and seamless container onboarding, a production-grade deployment would swap this to `validate` or `none` and utilize explicit migration scripts (like Flyway or Liquibase) to prevent accidental data modification.

## API Endpoints

| Method | Endpoint | Description | Status Code |
| :--- | :--- | :--- | :--- |
| GET | /api/posts | Fetch all posts (Paginated) | 200 OK |
| GET | /api/posts?term={term} | Search posts by title, content, or category | 200 OK |
| GET | /api/posts/{id} | Fetch a single post by ID | 200 OK / 404 Not Found |
| POST | /api/posts | Create a new post | 201 Created / 400 Bad Request |
| PUT | /api/posts/{id} | Update an existing post | 200 OK / 404 Not Found |
| DELETE | /api/posts/{id} | Delete a resource | 204 No Content / 404 Not Found |

### API Resource Schema

The API handles the **Post/Article** resource using the following public schema structure. Fields marked as auto-generated are handled by the system and should not be sent in creation requests.

| Field Name  | Data Type    | Constraints                  | Description                              |
| :---        | :---         | :---                         | :---                                     |
| `id`        | String/Number| Auto-generated by system     | Unique identifier for the blog post      |
| `title`     | String       | Required | The title of the blog post               |
| `content`   | String       | Required                     | The main body text of the article        |
| `category`  | String       | Required                     | The classification category of the post  |
| `tags`      | List<String> | Required, Elements Not Blank | Associated tags for filtering posts      |
| `createdAt` | String (ISO) | Auto-generated by system     | Timestamp when the post was created      |
| `updatedAt` | String (ISO) | Auto-generated by system     | Timestamp of the last update             |

### Request Body (POST / PUT)

```json
{
  "title": "Mastering Spring Boot Containers",
  "content": "This is a detailed guide on containerizing Spring Boot apps...",
  "category": "Technology",
  "tags": ["springboot", "docker", "java"]
}
```

### Validation Constraints
The API enforces the following rules on the incoming request payload:

* **'title'**: String. **Required**. Cannot be empty or purely whitespace.
* **'content'**: String. **Required**. Cannot be empty or purely whitespace.
* **'category'**: String. **Required**. Cannot be empty or purely whitespace.
* **'tags'**: Array of Strings. **Required**. Must contain at least one tag, and individual tags within the list cannot be empty or purely whitespace.

### Error Handling
If a request fails validation, the API returns a `400 Bad Request` with a structured key-value map of the fields that failed validation and their respective failure reasons:

```json
{
  "title": "Title is required",
  "tags": "At least one tag is required"
}
```

## Running Tests

Because the final production Docker image uses a stripped-down JRE runtime to minimize footprint size, it does not contain the tooling required to execute the test suite. 

For fast feedback cycles during active development, tests should be executed locally on your host machine using the embedded Maven wrapper:

```bash
./mvnw test
```

> **Note**: This executes Controller, Service, and Repository slice tests against an isolated, in-memory H2 database configured in PostgreSQL mode, meaning you do not need Docker containers running to validate your code logic.
---

## Tearing Down
To stop the containers and remove the volumes (which wipes the database clean for a fresh start next time), run:

```bash
docker compose down -v
```
